## Summary

OpenClaw implements a **device pairing system** for WebSocket connections to the gateway. After clearing browser storage, the browser lost its paired device identity, triggering the "pairing required" (1008) error. The gateway has 3 pending pairing requests from the user's browser that must be approved before reconnecting.

## What is Pairing?

### Device Identity & Security

OpenClaw uses cryptographic device pairing for access control:

1. **Device ID**: SHA-256 hash derived from a device's public key
2. **Public Key**: Ed25519 public key generated by the client (browser/CLI)
3. **Request ID**: UUID for each pairing request
4. **Roles**: Assigned permissions (operator, agent, node)
5. **Scopes**: Granular capabilities (operator.admin, operator.pairing, etc.)

### Pairing States

- **Paired**: Device in `~/.openclaw/devices/paired.json` (currently empty: `{}`)
- **Pending**: Requests awaiting approval in `~/.openclaw/devices/pending.json` (3 requests found)
- **Auto-approved**: Local clients with `silent: true` skip manual approval

### When Pairing is Required

From `/app/dist/gateway/server/ws-connection/message-handler.js`:

```javascript
const paired = await getPairedDevice(device.id);
const isPaired = paired?.publicKey === devicePublicKey;
if (!isPaired) {
    const ok = await requirePairing("not-paired");
    if (!ok) {
        return; // Connection closed with "pairing required"
    }
}
```

**Trigger**: Browser cleared storage → lost device identity → gateway rejects as unpaired device

**Root Cause**: Container's local config doesn't have `gateway.remote.token` set to match `gateway.auth.token`.

## Pairing Workflow Architecture

### Normal Flow

```
1. Browser connects with device identity (publicKey + deviceId)
2. Gateway checks getPairedDevice(deviceId)
3. If paired: connection proceeds
4. If not paired:
   a. requestDevicePairing() creates pending request
   b. Gateway broadcasts "device.pair.requested" event
   c. Operator sees notification in UI
   d. Operator approves/rejects
   e. Gateway broadcasts "device.pair.resolved" event
   f. Client reconnects, now paired
```

### Auto-Approval Flow

```
1. Local client connects (isLocalClient = true)
2. Pairing request created with silent: true
3. Gateway immediately calls approveDevicePairing()
4. Gateway broadcasts "device.pair.resolved" event
5. Connection proceeds without rejection
```

### User's Broken Flow

```
1. Browser connects (remote IP, silent: false)
2. Pairing request created → pending.json
3. Gateway broadcasts "device.pair.requested" ✓
4. Connection closed (1008 pairing required) ✓
5. [MISSING] Operator approval via CLI/UI
6. [BLOCKED] Browser cannot reconnect
```

## Technical Notes

### Error Codes

- **1008**: WebSocket close code for policy violation (used for auth/pairing failures)
- **ErrorCodes.NOT_PAIRED**: Gateway error code for unpaired device

### Security Model

- **Public key cryptography**: Ed25519 for device identity
- **Derived IDs**: SHA-256 hash prevents device ID spoofing
- **Role-based access**: operator > agent > node hierarchy
- **Scope-based permissions**: Granular capabilities per device

### Broadcast Events

- `device.pair.requested`: New pairing request (all operators notified)
- `device.pair.resolved`: Pairing approved/rejected (all clients notified)

### Threading

Device pairing uses file-based locking (`withLock()`) to prevent race conditions during concurrent approve/reject operations.
